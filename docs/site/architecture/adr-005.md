# ADR-005: Runtime observability is a required companion to cognitive and extension behavior

**Date:** 2026-04-10T19:50:15.145Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_cognitive_engine, data_model_narrative, data_model_tensions  

## Context

The system exposes many adaptive and stateful behaviors, but without runtime metrics it is difficult to validate tool reliability, cognitive strategy quality, or extension/backend failure patterns.

**Constraints:**
- Observability should not block local development
- Metrics should help explain behavior rather than merely count requests
- The system includes both backend cognitive flows and UI client interactions

## Decision

Adopt runtime observability as a first-class operational requirement. Tool execution, cognitive transitions, schedule outcomes, and extension/backend failures should emit structured metrics or logs consumable by a configured runtime endpoint or local fallback sink.

### Alternatives Considered

- **Rely only on static analysis and git history** — Rejected: These cannot reveal production/runtime failure modes or usage patterns.
- **Add telemetry only for backend HTTP requests** — Rejected: This misses extension UX, cognitive scheduling, and internal tool behavior.

## Consequences

- Faster diagnosis of failure hotspots
- Better calibration of cognitive strategies based on live evidence
- Improved understanding of extension reliability and degraded modes

**Risks accepted:**
- Instrumentation adds implementation effort
- Poorly scoped telemetry could create noise if not curated

## Guard Rails

> ⚠️ Do NOT add major new cognitive or extension workflows without considering how they will be observed in runtime.

> ⚠️ Do NOT treat absent runtime data as evidence that a feature is healthy.

> ⚠️ Do NOT couple observability to a single external endpoint without a local fallback or graceful no-op mode.
