# ADR-002: Discipline sessions are mandatory governance for risky code modification workflows

**Date:** 2026-04-10T19:50:15.094Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** data_model_discipline_sessions, dreamgraph_src_discipline_types  

## Context

The repository includes a discipline subsystem, but there is no ADR documenting why modification tasks should be structured into evidence-backed phases with write protections.

**Constraints:**
- The project is modified by humans and agents
- Some tools can write source files and persistent knowledge
- Ground-truth evidence is required before proposing or executing changes
- Future contributors need clear expectations about when planning and verification are mandatory

## Decision

Use discipline sessions as the default governance mechanism for audits, ports, reconstructions, and non-trivial modifications. Evidence collection in INGEST/AUDIT, explicit planning, and post-change verification are the intended operating model for risky changes.

### Alternatives Considered

- **Allow free-form modification without phase gating** — Rejected: This increases the risk of unsupported claims, unsafe writes, and silent regressions.
- **Use discipline only for external contributors** — Rejected: Internal and agent-driven changes have the same need for traceable evidence and guarded execution.

## Consequences

- Safer write operations and better auditability
- Improved reproducibility of agent-assisted changes
- More explicit link between source-of-truth evidence and implemented diffs

**Risks accepted:**
- The workflow adds overhead for small changes
- Tooling bugs in the discipline layer can block legitimate work

## Guard Rails

> ⚠️ Do NOT weaken evidence requirements for risky code changes without a replacement governance mechanism.

> ⚠️ Do NOT permit protected writes in modification workflows without an approved plan or equivalent explicit review policy.

> ⚠️ Do NOT remove verification expectations for changes that alter behavior, persistence, or user-visible flows.
