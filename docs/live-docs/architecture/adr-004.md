# ADR-004: Chat conversation state in the VS Code extension is owned by the extension host and must survive webview recreation

**Date:** 2026-04-10T19:50:15.127Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** dreamgraph_extensions_vscode_src_types  

## Context

The chat UI can be torn down when users switch away and return, causing user-visible loss of conversation content if state depends on transient webview DOM state.

**Constraints:**
- VS Code webviews may be hidden and recreated by the host
- Users expect chat continuity within a workspace and instance
- The extension already includes chat memory facilities and instance-aware behavior

## Decision

Keep the extension host as the authoritative owner of chat state and rehydrate any recreated or re-shown webview from host-held memory. Persist conversation history per DreamGraph instance so view switches and panel recreation do not erase the chat transcript.

### Alternatives Considered

- **Store chat transcript only in the webview DOM or JavaScript runtime** — Rejected: This state is lost when the webview is disposed or reloaded.
- **Treat each chat panel reveal as a fresh session** — Rejected: This breaks conversational continuity and frustrates users.

## Consequences

- Switching tabs/views will no longer erase the chat history
- Extension-host logic remains the single source of truth for message history
- Instance-aware chat restoration becomes deterministic

**Risks accepted:**
- Persisted history may need pruning or explicit clearing UX
- Hydration logic must avoid duplicate rendering when panels are revived

## Guard Rails

> ⚠️ Do NOT rely on transient webview memory as the only storage for chat transcript state.

> ⚠️ Do NOT clear host-side chat state merely because the panel is hidden or recreated.

> ⚠️ Do NOT add a second authoritative transcript store without defining reconciliation rules.
