# ADR-001: Cognitive lifecycle state machine is the authoritative orchestration model

**Date:** 2026-04-10T19:50:15.079Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_cognitive_engine, feature_dream_cycle, feature_dream_normalization, workflow_dream_cycle, data_model_discipline_sessions  

## Context

The system contains multiple cognitive modes and transitions, but no recorded ADR explains why the lifecycle exists or what invariants future changes must preserve.

**Constraints:**
- Cognitive operations are stateful and may have long-running side effects
- Dreaming, normalization, lucid exploration, nightmare scanning, and discipline ingest must remain distinguishable
- Future changes by humans or agents need guard rails to avoid invalid state transitions

## Decision

Treat the explicit cognitive lifecycle states (including AWAKE, REM, NORMALIZING, NIGHTMARE, LUCID, and INGEST) as the authoritative orchestration model. New cognitive actions must either map onto an existing lifecycle state or extend the lifecycle with a reviewed ADR.

### Alternatives Considered

- **Use ad hoc booleans and per-feature execution flags** — Rejected: This would fragment orchestration logic and make transition safety unverifiable.
- **Collapse all non-awake behavior into a single background state** — Rejected: This would erase important semantic differences between dreaming, normalization, adversarial scanning, and interactive lucid investigation.

## Consequences

- State transitions become a protected architectural seam
- Tooling and docs can reason consistently about system behavior
- New cognitive features must explicitly define how they enter and leave lifecycle states

**Risks accepted:**
- The lifecycle may feel heavyweight for experimental features
- Adding new states requires deliberate governance

## Guard Rails

> ⚠️ Do NOT add a new cognitive execution mode without mapping it to the lifecycle or recording a replacement ADR.

> ⚠️ Do NOT bypass lifecycle transitions in tooling or extension integrations.

> ⚠️ Do NOT merge lifecycle states solely for implementation convenience without reviewing downstream tool and prompt impacts.
