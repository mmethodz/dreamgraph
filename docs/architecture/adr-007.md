# ADR-007: Preserve end-to-end agentic tool execution loop in VS Code chat panel

**Date:** 2026-04-16T17:57:55.880Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_agentic_tool_execution_loop, feature_architect_tool_call_translation, workflow_architect_agentic_tool_execution  

## Context

The VS Code chat experience could narrate intended tool use without actually executing tools when the chat panel streamed placeholder text instead of running the architect LLM tool loop. This severed the end-to-end loop between model-emitted tool_use instructions and actual local/MCP tool execution, producing false success narratives and unreliable operational behavior.

**Constraints:**
- All tool-capable chat responses must route through the architect LLM tool orchestration path rather than a placeholder streaming path.
- The chat panel must preserve iterative execution semantics: model emits tool_use -> system executes tool -> tool_result is appended -> model continues until final answer.
- Both MCP tools and local support tools are part of the supported execution surface and must remain invocable from the same loop.
- User-visible success claims must be grounded in actual executed tool results, not inferred intent or narrated plans.
- Changes must stay within the VS Code extension project boundary and avoid direct mutation of protected graph data outside approved MCP tools.

## Decision

The VS Code chat panel must use the architect LLM's tool-capable call path as the authoritative execution mechanism for chat interactions that may invoke tools. The system must maintain a closed-loop agentic workflow: gather tool definitions, invoke callWithTools, detect tool_use outputs, dispatch each tool call to the appropriate executor (local support tool or MCP tool), append structured tool_result messages, and continue iterating until the model returns a final answer without pending tool calls.

### Alternatives Considered

- **Keep placeholder stream()-based chat rendering and treat tool mentions as informational only** — Rejected: This creates a misleading user experience where the assistant appears to act without executing tools, breaking trust and preventing real orchestration.
- **Allow separate ad hoc execution paths for local tools and MCP tools** — Rejected: Multiple execution paths increase drift risk, make verification harder, and can break the required tool_result continuation semantics.
- **Rely on manual operator verification instead of enforcing execution-loop integrity in code** — Rejected: Manual verification can catch regressions after the fact but does not prevent false success claims or broken automation at runtime.

## Consequences

- Chat responses that mention tool activity will be backed by actual tool executions.
- The architect LLM layer remains the single authoritative translation point for provider-specific tool-call formats.
- Future regressions are easier to detect because the execution chain is conceptually explicit: tool_use -> execute -> tool_result -> continue.
- The extension can safely support mixed local and MCP orchestration in one conversation loop.

**Risks accepted:**
- The iterative tool loop is more complex than a plain streaming response path and may regress if refactored without preserving stop-reason handling.
- Provider-specific response-shape changes could break tool_use detection unless translation remains maintained.
- Error handling or rate-limit behavior may still cause partial executions if not surfaced clearly to the user.

## Guard Rails

> ⚠️ Do NOT replace the chat panel's tool-capable execution path with a placeholder or display-only streaming path for interactions that may invoke tools.

> ⚠️ Do NOT claim that a tool action succeeded unless a corresponding tool execution result was actually received and incorporated into the conversation.

> ⚠️ Do NOT split local-tool and MCP-tool orchestration into incompatible loops that bypass the shared tool_result continuation mechanism.

> ⚠️ Do NOT modify architect tool-call translation behavior without validating that emitted tool_use events still reach the chat-panel execution loop end-to-end.
