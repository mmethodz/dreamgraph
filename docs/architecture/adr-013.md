# ADR-013: Canonical UI Registry Uses Semantic ui_* IDs and Transitional Legacy Entries

**Date:** 2026-04-17T18:43:25.093Z  
**Status:** accepted  
**Decided by:** system  
**Affected entities:** feature_ui_registry, feature_dashboard_server, feature_vscode_extension, ui_dashboard_shell, dashboard_view, vscode_chat_panel, ui_status_bar, ui_context_inspector, ui_changed_files_view, ui_command_action_picker, ui_graph_signal_summary, ui_task_report_summary, ui_metrics_card_grid, ui_schedule_management_table, ui_docs_viewer  

## Context

The UI registry contains overlapping legacy and canonical entries for similar dashboard and extension surfaces, which causes exported docs and graph queries to mix authoritative semantic elements with older aliases or parallel representations.

**Constraints:**
- Preserve backward compatibility for existing registry consumers and older references.
- Do not delete legacy entries without a dedicated retirement mechanism.
- Canonical entries must remain abstract and platform-independent.
- Registry ownership should align with ADR-010 feature identity conventions.

## Decision

Treat semantically named ui_* entries and the established canonical dashboard/extension surfaces as authoritative. Legacy or parallel UI entries remain transitional references only and must not be preferred in new ownership mappings, new docs examples, or future enrichment passes.

### Alternatives Considered

- **Delete duplicate legacy UI entries immediately** — Rejected: There is no dedicated non-destructive retirement flow for UI elements, and immediate deletion risks breaking historical references.
- **Keep enriching all legacy and canonical entries equally** — Rejected: That would preserve ambiguity and continue to pollute the registry with overlapping authoritative representations.

## Consequences

- Future registry enrichment will target canonical ui_* entries first.
- Exported docs can be interpreted with a clear distinction between canonical and transitional UI entities.
- Ownership mappings and examples become more consistent across dashboard and extension surfaces.

**Risks accepted:**
- Legacy duplicate UI entries still remain queryable until a retirement mechanism exists.
- Some consumers may still surface legacy entries unless additional filtering is added in docs or queries.

## Guard Rails

> ⚠️ Do NOT create new authoritative UI registry entries for existing surfaces when a canonical ui_* entry already exists.

> ⚠️ Do NOT prefer legacy or parallel non-canonical UI entries in used_by mappings, examples, or enrichment passes when a canonical entry exists.

> ⚠️ Do NOT encode platform-specific styling tokens into canonical UI registry entries.
