# ADR-006: Architect chat must use structured tool-calling rather than stream-only narration

**Date:** 2026-04-16T17:39:18.338Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_vscode_extension, feature_agentic_tool_execution_loop, feature_architect_tool_call_translation, feature_mcp_server_runtime  

## Context

The VS Code architect chat panel delegated tool-capable requests to a placeholder that only streamed plain text. Because tool schemas were not sent to the provider API, the model narrated tool invocations as text/JSON instead of executing them, severing all MCP and local tool workflows.

**Constraints:**
- The VS Code architect must support both MCP tools and local extension tools
- Provider support must work across Anthropic and OpenAI-style tool-calling APIs
- Tool execution state must be represented as iterative assistant tool_use and user tool_result turns
- The extension must remain within the project workspace boundary and use existing local tool dispatch for filesystem and command execution

## Decision

Replace the stream-only placeholder path with a real iterative agentic loop that calls callWithTools, dispatches local or MCP tools based on the tool name, accumulates raw tool_use/tool_result messages, and continues until a non-tool final response is returned.

### Alternatives Considered

- **Keep plain streaming and describe tools in prompt only** — Rejected: This reproduces the failure mode where the model narrates intended actions without structured tool execution.
- **Implement provider-specific tool loops directly in the chat panel without using architect-llm callWithTools** — Rejected: This would duplicate logic already implemented in architect-llm and weaken cross-provider consistency.

## Consequences

- Architect responses can execute local and MCP tools instead of narrating them
- File creation and other action-oriented tasks become achievable through the chat UI
- Raw provider message translation remains centralized in architect-llm
- Future tool failures can be debugged at the orchestration boundary with clearer provenance

**Risks accepted:**
- Iterative tool loops can create runaway execution if stop reasons are mishandled
- Provider raw-message translation bugs can still cause partial failures
- Tool result handling remains sensitive to malformed tool outputs

## Guard Rails

> ⚠️ Do NOT route tool-capable architect requests through stream-only code paths.

> ⚠️ Do NOT bypass architect-llm callWithTools when structured tool execution is required.

> ⚠️ Do NOT replace iterative tool_use/tool_result accumulation with prompt-only descriptions of tool availability.
