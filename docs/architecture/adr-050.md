# ADR-050: Transport-layer exception normalization remains silent; logging occurs at context-rich boundaries

**Date:** 2026-04-22T08:02:01.386Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** dreamgraph_src_utils, dreamgraph_src_server, dreamgraph_src_mcp, workflow_dream_cycle  

## Context

The system needed an explicit rule for whether transport-layer exception normalization helpers such as safeExecute should automatically log caught exceptions, or whether logging should occur only at higher-level handlers that have operation context. Without a decision, future changes risk duplicate logs, noisy stderr output near MCP transport, and blurred separation of concerns between error shaping and operational observability.

**Constraints:**
- stdout is reserved for MCP JSON-RPC transport and operational logs must go to stderr
- logger behavior is deliberately sanitized/redacted in src/utils/logger.ts
- safeExecute in src/utils/errors.ts currently converts thrown exceptions into structured ToolResponse errors without logging
- no accepted ADR previously governed centralized exception logging at the transport normalization boundary
- logging quality depends on having tool/operation context, severity, and safe metadata at the call site

## Decision

Keep transport-layer exception normalization helpers silent by default. Helpers such as safeExecute may normalize uncaught exceptions into protocol-safe structured responses, but they must not automatically emit logs merely because they caught an exception. Logging should occur at context-rich boundaries such as tool handlers, server operations, or workflow orchestration points where the system knows what operation failed, what severity applies, and what metadata is safe and useful to record.

### Alternatives Considered

- **Always log every exception inside safeExecute** — Rejected: Would likely cause duplicate logging, produce low-context error lines, and increase stderr noise near the transport boundary.
- **Never log exceptions anywhere and rely only on structured error returns** — Rejected: Would weaken operational observability and make production troubleshooting harder when callers do not emit their own logs.
- **Make transport normalization log by default but allow callers to suppress it** — Rejected: Would invert the preferred ownership model and still encourage low-context logging at the wrong architectural boundary.

## Consequences

- Preserves separation of concerns between protocol-safe error shaping and operational logging
- Encourages logging decisions where full operation context is available
- Reduces duplicate or noisy stderr output
- Makes future centralized logging changes require explicit architectural review rather than incremental drift

**Risks accepted:**
- Some callers may forget to log important failures, reducing observability
- Silent normalization may hide stack/context unless higher-level handlers add logging
- Developers may incorrectly assume safeExecute logs failures unless documentation and prompts reinforce the rule

## Guard Rails

> ⚠️ Do NOT add automatic logging to safeExecute or equivalent transport-layer normalization helpers without revisiting this ADR.

> ⚠️ Do NOT treat exception normalization as sufficient observability for operationally significant failures; log at the handler/orchestration layer when context exists.

> ⚠️ Do NOT emit logs from transport-layer helpers that lack operation context, severity classification, or safe metadata merely because an exception was caught.

> ⚠️ Revisit this ADR before introducing centralized exception logging that changes stderr volume or duplicate-log behavior.
