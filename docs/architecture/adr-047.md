# ADR-047: Two-pass graph context fetch with confidence-gated direct feature refinement

**Date:** 2026-04-20T21:27:56.110Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** graph-relevance-propagation, workflow_vscode_assisted_reasoning, symbol-bounded-excerpt  

## Context

File-path-only graph context retrieval produced useful but lower-confidence feature matches, while exact feature_id matches produced higher relevance. However, canonical feature IDs are not available until after Phase 3 anchor promotion, which occurs after the initial graph context fetch in ContextBuilder.buildEnvelope.

**Constraints:**
- Keep all operations within the VS Code extension and DreamGraph daemon project boundary
- Preserve existing one-round-trip behavior when no strong feature candidate exists
- Avoid requiring full architectural reordering of buildEnvelope phases
- Use daemon /api/graph-context exact feature_ids matching when a strong candidate is available
- Retain backward-compatible fallback behavior if the second pass fails or returns no new data

## Decision

Use a two-pass graph context fetch inside _resolveGraphContext. Pass 1 always queries by file_path and evidence flags. Then a local confidence-gated pre-scorer evaluates Pass-1 features against the active anchor identity. When a candidate score is >= 0.75, Pass 2 re-queries /api/graph-context with feature_ids:[candidateId] to trigger the daemon's exact-match path and higher per-entity relevance. Merge Pass-2 entities into the existing graph context, de-duplicating by ID and upgrading relevance in place. If no strong candidate exists or Pass 2 fails, retain the Pass-1 result unchanged.

### Alternatives Considered

- **Single-pass file_path-only retrieval** — Rejected: It cannot reliably exploit exact feature_id matching, so relevance remains lower and less anchor-specific.
- **Reorder buildEnvelope so Phase 3 promotion happens before graph fetch** — Rejected: This would require broader architectural changes and promotion currently depends on graphContext candidates produced by the fetch itself.
- **Always run a second graph-context fetch** — Rejected: It adds unnecessary latency and cost on turns where no strong candidate exists.

## Consequences

- Higher feature relevance when the active anchor strongly maps to a returned feature
- Improved downstream evidence ordering because aggregateRelevance can observe direct-match scores
- No regression for weak or ambiguous anchors because the second pass is skipped
- A small latency increase only on high-confidence turns that trigger Pass 2

**Risks accepted:**
- Threshold sensitivity may under-trigger or over-trigger the second pass until tuned
- Future refactors may remove the second pass as redundant unless the rationale is preserved
- Merge logic could drift if graph context entity shapes change without corresponding updates

## Guard Rails

> ⚠️ Do NOT remove the confidence gate or unconditional fallback without re-evaluating latency versus relevance trade-offs

> ⚠️ Do NOT replace the two-pass merge with an unconditional second fetch on every turn without revisiting this ADR

> ⚠️ Do NOT assume canonical feature IDs are available before Phase 3 promotion unless buildEnvelope ordering is explicitly redesigned
