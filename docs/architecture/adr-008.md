# ADR-008: Use repository engine.env template for new instance creation with programmatic fallback

**Date:** 2026-04-17T11:36:57.793Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_vscode_extension, extensions_vscode  

## Context

New DreamGraph instances currently generate config/engine.env programmatically inside instance lifecycle code, while a repository template also exists under templates/default/config. This duplicates configuration defaults, risks drift when new environment variables are introduced, and makes onboarding documentation harder to keep correct. Instance creation should prefer a human-maintained default template when available, while remaining robust if the template is missing.

**Constraints:**
- New instance creation must continue to succeed even if the repository template is unavailable.
- The default engine.env must live in the repository templates/default/config directory and may also be mirrored into the global ~/.dreamgraph/templates/default/config directory.
- New engine.env variables should be represented in the default template as commented-out entries so users can discover all configurable options.
- Per-instance config/engine.env remains an override layer loaded by src/utils/engine-env.ts at startup.
- Changes must stay within the current instance lifecycle architecture and not break existing dg init behavior.

## Decision

When creating a new instance, DreamGraph must first attempt to copy config/engine.env from the repository template path templates/default/config/engine.env (and future compatible template locations if introduced). If no template file is available, it must fall back to generating engine.env programmatically as it does today. The repository template becomes the primary source of truth for the default commented-out engine.env content. Future introduction of new engine environment variables must include updating this default template so newly created instances expose them discoverably.

### Alternatives Considered

- **Keep generating engine.env only in code** — Rejected: This duplicates defaults, hides available variables from maintainers, and causes documentation and onboarding drift when new variables are added.
- **Require template presence and fail instance creation if missing** — Rejected: This would make dg init brittle and regress current reliability guarantees.
- **Use only a global ~/.dreamgraph template outside the repository** — Rejected: Repository-controlled defaults are needed for versioned source-of-truth behavior and contributor visibility.

## Consequences

- New instances will inherit a single maintained default engine.env template when available.
- Instance creation remains reliable because programmatic fallback still works.
- Adding new engine env variables will require template maintenance as part of the change.
- Documentation and generated defaults will stay easier to align with actual onboarding behavior.

**Risks accepted:**
- Template and fallback code may still drift if maintainers forget to update both when new variables are introduced.
- Multiple template locations could become inconsistent if not intentionally synchronized.

## Guard Rails

> ⚠️ Do NOT remove the programmatic engine.env fallback unless template availability is guaranteed and migration is planned.

> ⚠️ Do NOT introduce new engine environment variables without updating the default engine.env template content to include them as commented-out examples.

> ⚠️ Do NOT change new-instance engine.env creation behavior to depend exclusively on user-specific global template files.
