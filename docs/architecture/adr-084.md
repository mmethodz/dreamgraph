# ADR-084: Consolidate existing VS Code extension UI registry entries under generic canonical IDs

**Date:** 2026-04-26T23:51:30.667Z  
**Status:** accepted  
**Decided by:** system  
**Affected entities:** dashboard_view, changed_files_view, graph_signal_indicator, context_inspector_output, dreamgraph_chat_panel, dreamgraph_tool_progress_view, dreamgraph_recommended_actions_bar, dreamgraph_status_bar_indicator  

## Context

The UI registry contains a mixed naming scheme where some VS Code extension UI elements use extension-prefixed ids while others use generic semantic ids. ADR-083 established generic semantic ids as canonical, but existing prefixed entries still create duplicate identity and ambiguity.

**Constraints:**
- Must follow ADR-083, which makes generic semantic identifiers canonical for UI registry entries.
- Must preserve semantic identity of existing UI concepts without losing VS Code implementation provenance.
- Must avoid introducing new platform-specific canonical ids when a generic semantic concept already exists.
- Registry consolidation should map extension-specific entries into canonical generic equivalents rather than maintaining duplicate top-level identities.

## Decision

Treat the existing generic semantic ids as the canonical UI registry identities and consolidate extension-prefixed entries into the following generic ids: dreamgraph_chat_panel -> chat_panel, dreamgraph_tool_progress_view -> tool_progress_view, dreamgraph_recommended_actions_bar -> recommended_actions_bar, dreamgraph_status_bar_indicator -> status_bar_indicator. VS Code-specific details remain in implementations metadata rather than in canonical ids.

### Alternatives Considered

- **Keep both generic and extension-prefixed ids active indefinitely** — Rejected: Preserves ambiguity, weakens canonical lookup, and conflicts with ADR-083 guard rails.
- **Make all ids extension-prefixed** — Rejected: Encodes implementation platform/product into identity and reduces portability across platforms.

## Consequences

- UI registry lookups become more consistent because each UI concept has one canonical semantic id.
- Future platform implementations can attach to the same semantic ids without renaming the concept.
- Existing references to extension-prefixed ids may require a compatibility/migration layer in consumers if they were used directly.

**Risks accepted:**
- If downstream tooling references extension-prefixed ids directly, consolidation may create stale references until migrated.
- Without a dedicated registry-rename operation in this tool surface, policy may be ahead of physical registry mutation.

## Guard Rails

> ⚠️ Do not create new canonical UI ids for VS Code extension elements using a dreamgraph_ or other platform/product prefix when a generic semantic id exists.

> ⚠️ Use generic semantic ids as canonical identities and store VS Code specificity only in implementations metadata.

> ⚠️ When consolidating duplicate UI elements, migrate references toward the generic canonical id rather than preserving both as equal peers.
