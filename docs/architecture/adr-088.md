# ADR-088: DreamGraph Explorer unresolved operational tradeoffs

**Date:** 2026-04-27T22:46:49.439Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** dreamgraph-explorer, explorer-live-observe, explorer-curated-mutation, graph-event-stream, explorer-graph-snapshot  

## Context

DreamGraph Explorer has accepted safety guard rails, but several operational tradeoffs remain unresolved: legacy-mode authorization is weaker than strict instance isolation, bounded SSE replay creates replay-gap UX/reset costs, and optimistic concurrency via If-Match can create retry friction on high-churn views. These issues should be made explicit so future changes evaluate them deliberately rather than rediscovering them implicitly.

**Constraints:**
- ADR-087 already establishes Explorer's binding consistency and mutation-safety model.
- The current tension log does not contain explicit Explorer-specific unresolved tradeoff records.
- Explorer must continue to support snapshot-first browsing, SSE live updates, and curated mutation flows.
- Operational tradeoffs should be visible without weakening accepted guard rails.

## Decision

Record the unresolved Explorer operational tradeoffs as an explicit governance artifact so they remain visible: legacy-mode auth weakness vs deployment simplicity, bounded SSE replay vs lightweight transport, and strict If-Match concurrency vs user retry friction. Future Explorer changes should evaluate these tradeoffs against ADR-087 guard rails instead of relaxing the guard rails silently.

### Alternatives Considered

- **Leave the tradeoffs implicit in code and ADR risk text only** — Rejected: Important unresolved concerns would be harder to discover and easier to regress during later feature work.
- **Relax the existing Explorer guard rails now to reduce friction** — Rejected: That would trade away safety and consistency without a dedicated replacement design.
- **Wait for production incidents before documenting these tradeoffs** — Rejected: Reactive documentation would lose architectural context and delay mitigation planning.

## Consequences

- Explorer governance now distinguishes binding guard rails from unresolved operating tensions.
- Future work can explicitly target legacy auth hardening, replay-gap UX, and stale-etag ergonomics.
- Teams reviewing Explorer changes have a stable reference for known tradeoffs.

**Risks accepted:**
- This ADR records tensions but does not itself resolve them.
- Some users may interpret the documented tradeoffs as defects even when they are accepted temporary compromises.
- Additional follow-up tracking may still be needed if operational evidence grows.

## Guard Rails

> ⚠️ Do NOT remove or weaken ADR-087 guard rails when addressing Explorer operational tradeoffs; tradeoff mitigation must preserve snapshot-first consistency and curated mutation safety.

> ⚠️ Do NOT treat legacy-mode authorization, replay-gap resets, or stale-etag retries as solved without explicit replacement design and validation.

> ⚠️ Do NOT optimize Explorer UX by bypassing audit, instance isolation, or optimistic concurrency requirements without a superseding ADR.
