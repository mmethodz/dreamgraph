# ADR-051: Programmatic config and seed-data fallbacks are emergency stubs, not feature copies

**Date:** 2026-04-24T19:36:16.540Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** instance_lifecycle, engine_env, data_stubs, templates_default  

## Context

ADR-008 required a programmatic fallback in src/instance/lifecycle.ts so `dg init` never hard-fails when `templates/default/config/engine.env` is missing. The fallback was implemented as a ~100-line inline string mirroring the template variable-for-variable. The same pattern was reused for seed JSON files via DATA_STUBS in src/instance/types.ts, which duplicated the rich `_schema`/`_fields` documentation embedded in templates/default/*.json. The duplication satisfied ADR-008 literally but created a new failure mode: template and fallback could (and did) drift whenever a new env var or seed field was added on one side only. Recorded as F-11 in plans/SYSTEM_FINDINGS.md.

**Constraints:**
- ADR-008 Guard Rail #1 still applies — programmatic fallback must remain so `dg init` survives a missing template
- ADR-009's three-tier resolution (global → repo → programmatic) must remain intact; this ADR only narrows the third tier's content, not its existence
- The 19-file DATA_STUBS invariant asserted by tests/instance-isolation.test.ts must hold so legacy callers keep working
- Reduced fallbacks must still produce a writable, schema-valid instance — engine, dashboard, and CLI must function on stub output
- Tests must continue to pass without modification

## Decision

Reduce programmatic fallback content for both engine.env and DATA_STUBS to minimal emergency stubs: empty containers and a bare provider-selection comment. Rich documented defaults live exclusively in templates/default/. When the fallback path fires, lifecycle emits logger.warn identifying the missing template paths and instructing the user to reinstall. Concretely: engine.env fallback shrinks from ~100 lines to a 4-line stub naming DREAMGRAPH_LLM_PROVIDER and pointing at the template; DATA_STUBS keeps all 19 keys but drops every _schema/_fields/_note field, each value becoming the empty container shape; both fallback paths log at warn level (was debug).

### Alternatives Considered

- **Generate the fallback at build time from the template** — Rejected: Adds a build-time codegen step and a generated source file in src/, complicating the toolchain for marginal gain. May revisit if emergency stubs themselves start drifting.
- **Add a CI drift-detection test only** — Rejected: Keeps the duplication and defers the maintenance burden to PR reviewers — exactly the pattern that produced F-11 in the first place.
- **Read the template at runtime via fs.readFileSync resolved relative to dist/** — Rejected: This is what the existing copyFile(...) path already does; the fallback by definition runs only when that file is missing, so re-reading provides no value.
- **Replace the fallback with a hard error** — Rejected: Violates ADR-008 Guard Rail #1.

## Consequences

- Single source of truth for config/seed defaults is unambiguous: templates/default/
- New env vars and seed-file fields no longer require dual edits; only the template needs maintenance, satisfying F-11
- Operators get a loud, actionable warning when a broken install causes the fallback path to fire instead of a silent debug line
- Instances created from the emergency fallback contain less discoverable documentation than template-created instances — intentional and recoverable via reinstall
- ADR-008 and ADR-009 are refined, not superseded; their decisions about resolution order, fallback presence, and named-template support are unchanged

**Risks accepted:**
- An instance bootstrapped purely from the emergency stub will have empty seed files without _schema annotations; users encountering this state must reinstall to get documented defaults — the warn log says so
- DATA_STUBS is still hand-maintained; if the 19-file invariant changes, both DATA_STUBS and templates/default/ need an entry, plus the test count assertion. Narrower drift surface than full-content duplication but not zero.

## Guard Rails

> ⚠️ Do NOT re-expand the in-code engine.env fallback or DATA_STUBS entries to mirror template content. Their job is to keep `dg init` working, not to duplicate the template.

> ⚠️ Do NOT downgrade the fallback logger.warn calls to debug or remove them. They are the only signal that a broken install caused fallback usage.

> ⚠️ Do NOT remove the programmatic fallback path entirely — ADR-008 Guard Rail #1 still applies.

> ⚠️ When adding a new seed file, update both templates/default/<file>.json (rich, documented) and DATA_STUBS (minimal empty container) and adjust the tests/instance-isolation.test.ts count assertion.
