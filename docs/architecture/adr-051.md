# ADR-051: Programmatic config and seed-data fallbacks are emergency stubs, not feature copies

**Date:** 2026-04-24T00:00:00.000Z
**Status:** accepted
**Decided by:** collaborative
**Affected entities:** instance_lifecycle, engine_env, data_stubs, templates_default
**Refines:** ADR-008, ADR-009

## Context

ADR-008 established that `templates/default/config/engine.env` is the source of
truth for new-instance configuration and required a programmatic fallback so
`dg init` never hard-fails when the template is missing. The fallback was
implemented in [`src/instance/lifecycle.ts`](../../src/instance/lifecycle.ts) as a
~100-line inline string array that mirrored the template content variable for
variable. The same pattern was reused for seed JSON files via
[`DATA_STUBS`](../../src/instance/types.ts) in `src/instance/types.ts`, which
duplicated the rich `_schema`/`_fields` documentation embedded in
`templates/default/*.json`.

The duplication satisfied ADR-008 literally — a fallback existed — but
created a new failure mode the ADR did not address: the template and the
fallback could (and historically did) drift whenever a new env var or seed
field was added on one side only. F-11 in `plans/SYSTEM_FINDINGS.md` recorded
this as a config-generation duplication smell.

**Constraints:**

- ADR-008 Guard Rail #1 still applies — the programmatic fallback must remain
  so `dg init` survives a missing template.
- ADR-009's three-tier resolution (`global → repo → programmatic`) must remain
  intact; this ADR only narrows the third tier's *content*, not its existence.
- The 19-file `DATA_STUBS` invariant asserted by
  `tests/instance-isolation.test.ts` must hold so legacy callers and other
  tooling that import the constant keep working.
- Reduced fallbacks must still produce a writable, schema-valid instance —
  cognitive engine startup, dashboard, and CLI must function on the stub
  output even if the user never installs the templates.
- Tests must continue to pass without modification.

## Decision

The programmatic fallback content for both `engine.env` and `DATA_STUBS` is
reduced to **minimal emergency stubs**: empty containers and the bare
provider-selection comment. Rich, documented defaults live exclusively in
`templates/default/`. When the fallback path is taken, the lifecycle emits a
`logger.warn` identifying the missing template paths and instructing the user
to reinstall — making the silent-drift failure mode loud and recoverable.

Concretely:

- `engine.env` fallback shrinks from ~100 lines of commented variables to a
  4-line stub naming `DREAMGRAPH_LLM_PROVIDER` and pointing at the template.
- `DATA_STUBS` keeps all 19 keys but drops every `_schema`, `_fields`, and
  `_note` field. Each value becomes the empty container shape (e.g. `[]`,
  `{ entities: {} }`, `{ chapters: [], weekly_digests: [], trends: {} }`).
- Both fallback paths log at `warn` level (was `debug`) so a broken install
  surfaces in normal operator output instead of silent debug logs.

### Alternatives Considered

- **Generate the fallback at build time from the template** — Rejected for
  this iteration: adds a build-time codegen step and a generated source file
  in `src/`, which complicates the toolchain for a marginal gain. May revisit
  if the emergency stubs themselves start drifting.
- **Add a CI drift-detection test only** — Rejected: keeps the duplication and
  defers the maintenance burden to PR reviewers, exactly the pattern that
  produced F-11 in the first place.
- **Read the template at runtime via a `fs.readFileSync` resolved relative to
  `dist/`** — Rejected: this is what the existing `copyFile(...)` path
  already does; the fallback by definition runs only when that file is
  missing, so re-reading it provides no value.
- **Replace the fallback with a hard error** — Rejected: violates ADR-008
  Guard Rail #1.

## Consequences

- The single source of truth for config/seed defaults is now unambiguous —
  `templates/default/`.
- New env vars and seed-file fields no longer require dual edits; only the
  template needs maintenance, satisfying F-11.
- Operators get a loud, actionable warning when a broken install causes the
  fallback path to fire, instead of a silent `debug` line.
- Instances created from the emergency fallback contain less discoverable
  documentation than template-created instances. This is intentional and
  recoverable via reinstall.
- ADR-008 and ADR-009 are *refined*, not superseded; their decisions about
  resolution order, fallback presence, and named-template support are
  unchanged.

**Risks accepted:**

- An instance bootstrapped purely from the emergency stub will have empty
  seed files without `_schema` annotations. Users encountering this state
  must reinstall to get the documented defaults; the warning log says so.
- The `DATA_STUBS` constant is still hand-maintained. If the 19-file
  invariant changes (a new seed file is added), both `DATA_STUBS` and
  `templates/default/` need an entry. This is a much narrower drift surface
  than the previous full-content duplication.

## Guard Rails

> ⚠️ Do NOT re-expand the in-code `engine.env` fallback or `DATA_STUBS`
> entries to mirror template content. Their job is to keep `dg init` working,
> not to duplicate the template.

> ⚠️ Do NOT downgrade the fallback `logger.warn` calls to `debug` or remove
> them. They are the only signal that a broken install caused fallback usage.

> ⚠️ Do NOT remove the programmatic fallback path entirely — ADR-008 Guard
> Rail #1 still applies.

> ⚠️ When adding a new seed file, update both `templates/default/<file>.json`
> (rich, documented) and `DATA_STUBS` (minimal empty container) and adjust
> the `tests/instance-isolation.test.ts` count assertion.
