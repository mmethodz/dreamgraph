# ADR-014: UI Registry Supports Backward-Compatible Retirement and Export Filtering

**Date:** 2026-04-17T19:00:13.775Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_ui_registry, ui_dashboard_shell, ui_status_bar, ui_task_report_summary, ui_graph_signal_summary, dashboard_view, vscode_chat_panel  

## Context

The UI registry contained canonical ui_* entries alongside legacy or parallel entries. Without lifecycle metadata, exports and queries surfaced all entries equally, making the semantic inventory noisy and making canonicalization incomplete.

**Constraints:**
- Must remain backward compatible with existing ui_registry.json data and register_ui_element payloads.
- Must not delete historical UI entries destructively.
- Canonical ui_* entries established by ADR-013 must remain authoritative.
- Living docs should prefer active canonical entries while still preserving historical traceability.

## Decision

Add optional UI registry lifecycle fields (status, superseded_by, deprecation_reason). Treat omitted status as active for backward compatibility. Extend query/export behavior so deprecated entries can be filtered from primary views while still being documented as legacy references.

### Alternatives Considered

- **Delete legacy UI entries outright** — Rejected: Would lose historical traceability and risks breaking existing references.
- **Keep legacy entries without lifecycle metadata** — Rejected: Leaves canonicalization ambiguous and docs noisy.

## Consequences

- Canonical UI entries become easier to identify in queries and exports.
- Legacy entries remain available for compatibility but no longer dominate primary documentation.
- Future retirement of overlapping UI entries can be done non-destructively.

**Risks accepted:**
- Some existing exports may show newly added deprecated sections, changing documentation shape slightly.
- Legacy entries without explicit superseded_by values would remain ambiguous until curated.

## Guard Rails

> ⚠️ Do NOT require lifecycle fields on older UI registry entries; omitted status must continue to mean active.

> ⚠️ Do NOT delete legacy UI entries when a non-destructive deprecated or transitional state is sufficient.

> ⚠️ Do NOT include deprecated UI entries in primary migration planning by default.

> ⚠️ Do NOT let deprecated legacy entries outrank canonical ui_* entries in exported documentation.
