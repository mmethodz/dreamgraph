# ADR-046: Semantic anchors are canonical; positional coordinates are advisory only

**Date:** 2026-04-20T20:37:04.091Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** workflow_vscode_assisted_reasoning  

## Context

Context anchoring based on line numbers, cursor positions, and offsets is brittle under edits, symbol movement, and refactors. The extension needs a stable identity model for evidence selection, packet rendering, and cross-session context reuse.

**Constraints:**
- Coordinates can drift after edits and must not be treated as identity.
- Anchor resolution must prefer semantic identifiers over positional hints.
- VS Code symbol resolution is the primary local authority before heuristic fallback.
- Legacy coordinate-based anchors must remain backward-compatible during migration.
- Graph-backed canonical IDs should be used when available for promotion across features, workflows, ADRs, and UI elements.

## Decision

Semantic anchors are the canonical identity for context. Preferred anchor hierarchy is graph entity IDs, symbol names/paths, workflow IDs, ADR IDs, UI registry IDs, relationship assertions, and semantic fingerprints. Repository paths, file paths, and module/class names are secondary. Line numbers, cursor positions, offsets, and snippets are advisory only and may be used as hints but never as canonical identity.

### Alternatives Considered

- **Continue treating line numbers and cursor positions as primary anchors** — Rejected: High drift rate under edits and poor cross-session stability cause stale evidence binding and degraded retrieval precision.
- **Use file paths plus line ranges as canonical identity** — Rejected: More stable than raw coordinates but still breaks under symbol movement, file splits, and refactors; does not capture semantic sameness.
- **Use heuristic/snippet-only anchoring** — Rejected: Useful as fallback, but ambiguous, token-expensive, and weaker than symbol or graph identity.

## Consequences

- Reasoning packets become more resilient to edits and symbol relocation.
- Stored anchors can be migrated by exact match, rebound on symbol moves, downgraded on drift, or archived when no match exists.
- Evidence selection can prioritize relevance, confidence, token cost, and anchor relation using canonical anchors.
- Rendered context will better communicate semantic identity and approximate matches.

**Risks accepted:**
- Symbol provider gaps or extraction misses may temporarily reduce anchor precision.
- Promotion to graph-backed canonical IDs depends on graph freshness and may be incomplete in sparse areas.
- Backward-compatible migration logic adds complexity during the transition period.

## Guard Rails

> ⚠️ Do NOT treat line numbers, cursor positions, offsets, or snippets as canonical identity in new context-building logic.

> ⚠️ Do NOT bypass VS Code symbol resolution when a symbol-aware anchor can be derived.

> ⚠️ Do NOT introduce full-file-read-first anchoring as the default path; full file reads remain planner-authorized, debugging-only, or explicitly requested.

> ⚠️ When a semantic anchor can be promoted to a graph-backed canonical ID, preserve that canonical identity through packet construction and rendering.

> ⚠️ Legacy anchors from prior sessions must be migrated through exact match, rebind, drift downgrade, or archive logic rather than trusted as-is.
