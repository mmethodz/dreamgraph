## Context

ADR-003 through ADR-007 were already accepted, but the graph lacked durable implementation anchors showing where their guard rails are concretely enforced in the current codebase. Without recorded anchors, future reasoning could drift back to generic architectural summaries instead of verified code surfaces.

**Constraints:**
- Must not alter the accepted meaning of ADR-003 through ADR-007
- Anchors must use current verified implementation surfaces in the VS Code extension host and orchestration layer
- Evidence must prefer semantic anchors over line numbers
- Recorded knowledge must stay within existing project scope and repo surfaces

## Decision

Record a companion architecture decision that captures the verified implementation anchors for ADR-003 through ADR-007 in the current codebase. ADR-003 is anchored in extension.ts, McpClient, and HealthMonitor. ADR-004 is anchored in ChatMemory plus ChatPanel host-side message ownership and rehydration. ADR-005 is anchored in HealthMonitor and the extension reporting/request-budget instrumentation path. ADR-006 is anchored in ArchitectLlm.callWithTools with provider-native tool schema translation. ADR-007 is anchored in the ChatPanel iterative tool execution loop and tool-result reinjection path.

### Alternatives Considered

- **Leave ADRs mapped only in transient chat history** — Rejected: This would lose the implementation mapping and weaken future graph-grounded reasoning.
- **Rewrite the original ADRs with code-level details** — Rejected: This would change accepted ADR content instead of recording verified implementation evidence as a separate durable artifact.

## Consequences

- Future architectural queries can reference both the normative ADRs and the verified implementation surfaces
- Graph-grounded reasoning about extension orchestration can cite concrete code anchors instead of restating ADR intent abstractly
- Drift between ADR intent and implementation should be easier to detect during later inspections

**Risks accepted:**
