# ADR-086: Use stateless Responses API for GPT-5.5 Architect initially

**Date:** 2026-04-27T14:40:07.824Z  
**Status:** accepted  
**Decided by:** human  
**Affected entities:** extensions/vscode/src/architect-llm.ts, architect-openai-gpt-55-responses-adapter, dreamgraph-architect-memory-context  

## Context

The GPT-5.5 Responses API migration guide recommends stateful context mechanisms such as previous_response_id for multi-turn workflows, but Architect already uses DreamGraph as the authoritative memory/context source and the newly added GPT-5.5 adapter still needs correctness hardening for tool calls before introducing stateful optimization complexity.

**Constraints:**
- DreamGraph remains the source of truth for memory and project context.
- Architect GPT-5.5 tool-call correctness must be stable before optimizing state handling.
- Avoid duplicating or diverging memory between OpenAI stored response state and DreamGraph graph/context assembly.
- Initial GPT-5.5 implementation scope covers non-streaming and tool-call paths, not full provider migration.

## Decision

Keep the Architect GPT-5.5 Responses API adapter stateless initially. Continue replaying relevant context and tool-call items explicitly. Defer previous_response_id/store-based statefulness to a later optional optimization layer after GPT-5.5 tool-call correctness is verified.

### Alternatives Considered

- **Adopt previous_response_id immediately for all GPT-5.5 Architect turns** — Rejected: Adds state coordination complexity before tool-call correctness is stable and risks creating a second memory source outside DreamGraph.
- **Use only Chat Completions until stateful Responses support is implemented** — Rejected: Conflicts with the GPT-5.5 migration guidance for reasoning, tool-calling, and multi-turn workflows, and would block the already implemented Responses API adapter.
- **Use OpenAI stored state as primary Architect memory** — Rejected: Violates the architectural principle that DreamGraph is the source of memory and context.

## Consequences

- GPT-5.5 Architect calls remain easier to reason about and debug because each request carries explicit context and tool replay items.
- DreamGraph continues to own long-term memory, graph context, decisions, workflows, and tensions.
- Latency and token cost may be higher than a future stateful Responses implementation because context is replayed manually.
- A later optimization can add previous_response_id/store once adapter correctness and observability are sufficient.

**Risks accepted:**
- Manual replay can increase prompt size and token cost for long conversations.
- Some Responses API stateful optimizations and cached context benefits are deferred.
- Future contributors may misinterpret absence of previous_response_id as an accidental migration gap unless this ADR is consulted.

## Guard Rails

> ⚠️ Do not make OpenAI previous_response_id/store the primary Architect memory source; DreamGraph remains authoritative.

> ⚠️ Do not add stateful Responses API handling for Architect until GPT-5.5 tool-call correctness has dedicated tests or equivalent verification.

> ⚠️ If previous_response_id/store is added later, it must be treated as an optimization/cache layer and must not replace DreamGraph context retrieval or graph updates.

> ⚠️ Keep stateless replay behavior available as a fallback even if stateful Responses support is introduced.
