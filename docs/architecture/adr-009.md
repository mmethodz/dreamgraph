# ADR-009: Support named global instance templates for dg init

**Date:** 2026-04-17T12:01:27.784Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_vscode_extension, extensions_vscode, workflow_architect_agentic_tool_execution  

## Context

New instances need a clear, reusable template mechanism so users can prepare multiple instance configuration presets under ~/.dreamgraph/templates and select one during dg init. Without an explicit architectural rule, template resolution and documentation can drift, and dg init help/docs may under-specify how template selection works.

**Constraints:**
- Must stay within the ~/.dreamgraph master directory boundary
- Must preserve programmatic fallback when no named template exists
- Must remain compatible with existing default template-first instance creation behavior from ADR-008
- Users, not the repo, are responsible for creating additional named templates such as openai or anthropic

## Decision

dg init supports --template <name> with default 'default'. Instance creation resolves templates in this order: ~/.dreamgraph/templates/<name> first, repository templates/<name> second, built-in programmatic scaffold last. Install scripts copy the repository templates/default tree into the global templates area and overwrite only with force or explicit user confirmation. Documentation and command help must describe that users can create additional templates by copying and renaming ~/.dreamgraph/templates/default.

### Alternatives Considered

- **Support only a hardcoded default template** — Rejected: Prevents user-managed presets and does not meet the requirement for reusable named templates such as openai.
- **Require all templates to live only in the repository** — Rejected: Makes per-user reusable presets awkward and undermines the purpose of ~/.dreamgraph/templates.
- **Remove programmatic fallback and require a template always exist** — Rejected: Reduces robustness and conflicts with ADR-008 resilience requirements.

## Consequences

- Users can prepare multiple reusable instance presets under ~/.dreamgraph/templates
- dg init help and docs become explicit about template selection and customization
- Default installation behavior keeps the global default template aligned with the repository template unless the user preserves a custom installation

**Risks accepted:**
- Template content may drift if new environment variables are added and templates are not updated
- User-customized templates may be overwritten if force/overwrite is used intentionally

## Guard Rails

> ⚠️ Do NOT remove the fallback chain: global template -> repository template -> programmatic scaffold without replacing it with an equally robust mechanism.

> ⚠️ Do NOT make dg init template selection implicit or undocumented; --template behavior must remain discoverable via dg init --help and setup docs.

> ⚠️ Do NOT assume named templates exist in the repository beyond default; users must be able to create them in ~/.dreamgraph/templates.
