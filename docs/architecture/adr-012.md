# ADR-012: UI Registry Supports Abstract Visual and Layout Semantics With Backward Compatibility

**Date:** 2026-04-17T18:33:52.592Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_ui_registry, ui_dashboard_shell, vscode_chat_panel  

## Context

The semantic UI registry captured purpose, data contracts, and interactions, but lacked first-class fields for abstract visual language and layout contracts. This limited reconstruction, migration planning, and design reasoning. The system needed these semantics without breaking existing registry entries or callers.

**Constraints:**
- Must preserve backward compatibility for existing ui_registry.json entries and existing register_ui_element callers
- Must keep the registry abstract and platform-independent
- Must not store raw CSS classes, Tailwind utilities, pixel spacing, or framework-specific style props as authoritative registry data
- Must continue supporting existing migration and query tools

## Decision

Extend the SemanticElement and register_ui_element schema with optional visual_semantics and layout_semantics fields, treat them as non-breaking enrichment, and bump the UI registry schema metadata to 1.1.0 while keeping older entries readable.

### Alternatives Considered

- **Keep encoding visual/layout intent only in tags and notes** — Rejected: Too weakly structured for reliable querying, migration planning, and architectural validation
- **Store raw implementation styling details in the registry** — Rejected: Violates abstraction boundaries and would couple the registry to specific UI stacks

## Consequences

- UI elements can now carry first-class abstract visual and layout metadata
- Existing registry entries remain valid without migration
- Future UI reasoning and migration planning can use stronger semantics

**Risks accepted:**
- Some existing consumers may ignore the new fields until updated
- Abstract semantics may be modeled inconsistently unless governed by ADR guard rails

## Guard Rails

> ⚠️ Do NOT require visual_semantics or layout_semantics for legacy UI entries to remain valid

> ⚠️ Do NOT store raw CSS classes, framework utility tokens, or pixel values in the UI registry as authoritative metadata

> ⚠️ Do NOT break existing register_ui_element payloads that omit the new fields
