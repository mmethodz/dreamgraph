# ADR-087: DreamGraph Explorer consistency and mutation safety model

**Date:** 2026-04-27T22:41:16.838Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** dreamgraph-explorer, explorer-readonly-browse, explorer-live-observe, explorer-curated-mutation, explorer-graph-snapshot, graph-event-stream  

## Context

DreamGraph Explorer introduces a live graph view, SSE event stream, and curated mutation endpoints. Without explicit guard rails, future changes could weaken instance isolation, break optimistic concurrency, or create ambiguous client consistency semantics between snapshot state and live events.

**Constraints:**
- Explorer serves both read-only graph browsing and curated mutation flows from the daemon HTTP surface.
- Live updates are delivered via SSE with bounded replay and heartbeat semantics.
- Mutation endpoints already require instance-scoped authorization, non-empty reason strings for governed intents, and optimistic concurrency via If-Match snapshot etags.
- Explorer must remain safe for graph/cognitive state by funnelling writes through curated intents with audit coverage rather than arbitrary graph mutation.

## Decision

Adopt a consistency model where Explorer clients bootstrap from a versioned graph snapshot and then layer live SSE events on top with bounded best-effort replay. If replay gaps exceed the event buffer, clients must refetch the snapshot and reset local derived state. Adopt a mutation safety model where Explorer writes remain curated-intent only, require instance authentication on mutation routes, require If-Match etag checks for optimistic concurrency where configured, and append audit rows for both success and failure.

### Alternatives Considered

- **Allow arbitrary direct Explorer graph writes without curated intents or audit** — Rejected: This would bypass governance and make cognitive/graph state changes hard to reason about or trace.
- **Use SSE as a fully durable source of truth with no required snapshot refetch on replay gaps** — Rejected: The event stream is backed by a bounded ring buffer and cannot guarantee infinite resumability.
- **Permit mutation endpoints without instance header or etag preconditions** — Rejected: This would weaken instance isolation and increase the risk of stale-write races from outdated Explorer clients.

## Consequences

- Explorer clients have a clear contract: snapshot first, then live SSE, and reset on replay gaps.
- Mutation flows stay auditable and bounded to approved intents.
- Future Explorer changes can evolve UI and transport details without changing the core safety model.
- Client implementations must handle gap/reset behavior and stale etag failures explicitly.

**Risks accepted:**
- SSE clients that ignore gap/reset semantics may show stale or divergent local state.
- Strict If-Match preconditions can increase retry friction for highly dynamic views.
- Legacy mode still allows mutation requests without UUID isolation, which reduces security guarantees in that mode.

## Guard Rails

> ⚠️ Do NOT treat the Explorer SSE stream as a complete durable event log; clients must be able to refetch graph snapshot state after replay gaps.

> ⚠️ Do NOT add arbitrary write endpoints under /explorer; Explorer mutations must remain curated intents routed through the mutation service with audit coverage.

> ⚠️ Do NOT remove instance-header checks from Explorer mutation routes without replacing them with an equivalent or stronger isolation mechanism.

> ⚠️ Do NOT remove If-Match optimistic concurrency from governed Explorer mutations without an explicit replacement concurrency strategy.

> ⚠️ Do NOT allow governed Explorer mutations to proceed without recording reason/audit context when the mutation definition requires it.
