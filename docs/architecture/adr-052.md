# ADR-052: Normalizer scoring yields to the event loop via setImmediate batching

**Date:** 2026-04-24T00:00:00.000Z
**Status:** accepted
**Decided by:** collaborative
**Affected entities:** cognitive_normalizer, dream_engine, daemon_responsiveness
**Related findings:** F-12

## Context

The cognitive normalizer in [`src/cognitive/normalizer.ts`](../../src/cognitive/normalizer.ts)
performs three linear passes over the dream graph during each normalization
cycle: PASS 1 (structural edge scoring), node validation, and PASS 3
(promotion gate). All three were synchronous `for...of` loops with no yield
points. For dream graphs in the 1 000+ edge range — which the cognitive
engine routinely produces after a handful of dream cycles — these loops
saturate the Node.js main event loop for hundreds of milliseconds at a time,
blocking incoming HTTP requests, dashboard polling, and other async work.
F-12 in `plans/SYSTEM_FINDINGS.md` flagged this as a P2 improvement and
suggested either `setImmediate()` batching (lightweight) or a worker thread
(heavier).

**Constraints:**

- The normalizer must remain in NORMALIZING engine state for the full pass
  (engine asserts this); the state machine cannot accommodate suspending
  and resuming across multiple async boundaries with foreign work in
  between.
- All three passes share the in-memory `lookup: FactLookup` and the
  `dreamGraph` mutation set; any threading model must preserve sequential
  consistency of edge/node status updates and the promotion gate's
  blocked-by-gate accounting.
- The 44-test invariant in `tests/instance-isolation.test.ts` must hold,
  and existing normalizer behaviour (counts, promotion outcomes, tension
  emission) must be unchanged for any input.
- The change must be reversible without a data migration.

## Decision

Insert `await new Promise(setImmediate)` yields every 100 iterations into
each of the three linear passes (edges, nodes, promotion gate). The yield
helper and the `NORMALIZER_YIELD_BATCH = 100` constant live at the top of
`src/cognitive/normalizer.ts`. The normalizer remains single-threaded; only
the event loop gets breathing room between chunks.

A worker-thread implementation is **explicitly deferred** behind a TODO
comment referencing this ADR. It would require serializing `FactLookup`
(Maps and Sets converted to plain arrays/objects) and a small worker-pool
module, neither of which is justified until profiling shows CPU saturation
dominating wall-clock time for normalization.

### Alternatives Considered

- **Move PASS 1 to a `worker_threads` pool now** — Rejected for this
  iteration: requires `FactLookup` serialization helpers, a pool
  abstraction, and lifecycle plumbing for cancellation/cleanup. Disproportionate
  cost for the current evidence (no reports of CPU-bound saturation).
- **Stream candidates through an async generator with implicit
  back-pressure** — Rejected: the three passes need full-batch
  cross-references (e.g. PASS 3 reads `edgeAssessments` produced by
  PASS 1; semantic boost runs over the whole list). A generator forces
  refactoring the multi-pass shape with no fairness gain over chunked
  yields.
- **Yield every iteration** — Rejected: per-iteration yields balloon the
  microtask queue and add measurable overhead with no responsiveness
  benefit beyond yielding every ~100.
- **Configurable batch size in cognitive tuning** — Deferred: 100 is a
  reasonable default and exposing another knob is premature. The constant
  can be promoted to tuning if a profile ever needs a different value.

## Consequences

- A normalization cycle over a 1 000-edge dream graph yields ~30 times
  across all three passes (≈10 + 10 + 10), letting the daemon service
  HTTP and dashboard traffic between chunks instead of blocking for the
  entire pass.
- Wall-clock time for a single normalization cycle increases marginally
  (one event-loop tick per yield, sub-millisecond each); throughput per
  cycle is essentially unchanged.
- The change is purely structural — no data migration, no behaviour
  change for any input, no new persistence.
- The TODO referencing this ADR creates a clear escalation path if
  profiling later shows CPU saturation: convert PASS 1 to a worker
  thread without revisiting the basic batching pattern.

**Risks accepted:**

- Concurrent mutations of `dreamGraph` from other code paths during the
  yield windows would now be observable. The engine's NORMALIZING state
  guard already prevents this (no other writer can run while the engine
  is in this state), but the assumption is now load-bearing in a way it
  wasn't before. If that guard is ever weakened, the yields become a
  correctness hazard.
- The 100-iteration constant is hand-tuned. Pathologically uneven edge
  validation costs (e.g. one extremely heavy LLM-augmented edge in a
  chunk) could still produce a noticeable pause inside a chunk. If
  observed, escalate to the worker-thread plan rather than shrinking
  the batch.

## Guard Rails

> ⚠️ Do NOT remove the `await yieldToEventLoop()` calls from the three
> passes. They are the F-12 fix; the daemon will block on large dream
> graphs without them.

> ⚠️ Do NOT mutate `dreamGraph`, `edgeAssessments`, or `lookup` from any
> other code path while normalization is running. The engine's NORMALIZING
> state guard enforces this and is now load-bearing for correctness.

> ⚠️ Do NOT shrink `NORMALIZER_YIELD_BATCH` below 25 or expand it above
> 500 without profiling evidence. The current value is a deliberate
> trade-off between fairness and overhead.

> ⚠️ When the worker-thread follow-up is implemented, update this ADR to
> "superseded" with a forward link to the new ADR. Do not silently delete
> the batching code — keep it as the single-thread fallback path.
