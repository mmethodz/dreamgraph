# ADR-052: Normalizer scoring yields to the event loop via setImmediate batching

**Date:** 2026-04-24T19:42:56.038Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** cognitive_normalizer, dream_engine, daemon_responsiveness  

## Context

The cognitive normalizer in src/cognitive/normalizer.ts performs three linear passes over the dream graph during each cycle: PASS 1 (structural edge scoring), node validation, and PASS 3 (promotion gate). All three were synchronous for...of loops with no yield points. For dream graphs in the 1 000+ edge range \u2014 routinely produced after a handful of dream cycles \u2014 these loops saturate the Node.js main event loop for hundreds of milliseconds, blocking incoming HTTP requests, dashboard polling, and other async work. Recorded as F-12 in plans/SYSTEM_FINDINGS.md (P2 improvement).

**Constraints:**
- Normalizer must remain in NORMALIZING engine state for the full pass; state machine cannot suspend/resume across foreign work
- All three passes share in-memory `lookup: FactLookup` and `dreamGraph` mutation set; sequential consistency of status updates and blocked-by-gate accounting must be preserved
- 44-test invariant in tests/instance-isolation.test.ts must hold; normalizer behaviour (counts, promotion outcomes, tension emission) unchanged for any input
- Change must be reversible without a data migration

## Decision

Insert `await new Promise(setImmediate)` yields every NORMALIZER_YIELD_BATCH=100 iterations into each of the three linear passes in src/cognitive/normalizer.ts (PASS 1 edge scoring, node validation, PASS 3 promotion gate). Yield helper and constant live at the top of the file. Normalizer remains single-threaded; only the event loop gets breathing room between chunks. Worker-thread implementation explicitly deferred behind a TODO referencing this ADR.

### Alternatives Considered

- **Move PASS 1 to a worker_threads pool now** — Rejected: Requires FactLookup serialization helpers, a pool abstraction, and lifecycle plumbing for cancellation/cleanup. Disproportionate cost without evidence of CPU saturation.
- **Stream candidates through an async generator** — Rejected: Three passes need full-batch cross-references (PASS 3 reads edgeAssessments produced by PASS 1; semantic boost runs over the whole list). Generator forces refactoring multi-pass shape with no fairness gain over chunked yields.
- **Yield every iteration** — Rejected: Per-iteration yields balloon the microtask queue and add measurable overhead with no responsiveness benefit beyond yielding every ~100.
- **Configurable batch size in cognitive tuning** — Rejected: 100 is a reasonable default; exposing another knob is premature. Constant can be promoted to tuning if a profile ever needs a different value.

## Consequences

- A normalization cycle over a 1 000-edge dream graph yields ~30 times across all three passes, letting the daemon service HTTP and dashboard traffic between chunks instead of blocking
- Wall-clock time per cycle increases marginally (one event-loop tick per yield, sub-ms each); throughput essentially unchanged
- Purely structural change — no data migration, no behaviour change for any input, no new persistence
- TODO referencing this ADR creates a clear escalation path if profiling later shows CPU saturation: convert PASS 1 to a worker thread without revisiting the basic batching pattern

**Risks accepted:**
- Concurrent mutations of dreamGraph from other code paths during yield windows would now be observable. The engine's NORMALIZING state guard prevents this, but the assumption is now load-bearing in a way it wasn't before. If that guard is ever weakened, the yields become a correctness hazard.
- The 100-iteration constant is hand-tuned. Pathologically uneven validation costs (e.g. one extremely heavy LLM-augmented edge in a chunk) could still produce a noticeable pause inside a chunk. If observed, escalate to the worker-thread plan rather than shrinking the batch.

## Guard Rails

> ⚠️ Do NOT remove the `await yieldToEventLoop()` calls from the three passes. They are the F-12 fix; the daemon will block on large dream graphs without them.

> ⚠️ Do NOT mutate dreamGraph, edgeAssessments, or lookup from any other code path while normalization is running. The engine's NORMALIZING state guard enforces this and is now load-bearing for correctness.

> ⚠️ Do NOT shrink NORMALIZER_YIELD_BATCH below 25 or expand it above 500 without profiling evidence. The current value is a deliberate trade-off between fairness and overhead.

> ⚠️ When the worker-thread follow-up is implemented, mark this ADR as superseded with a forward link to the new ADR. Do NOT silently delete the batching code — keep it as the single-thread fallback path.
