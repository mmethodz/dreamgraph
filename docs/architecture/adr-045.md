# ADR-045: Semantic anchors are canonical; line numbers and cursor positions are advisory only

**Date:** 2026-05-01T00:00:00.000Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_semantic_anchor_promotion, feature_context_builder, workflow_vscode_assisted_reasoning  

## Context

The DreamGraph VS Code extension previously used line numbers, cursor positions, and file-path offsets as the primary way to identify what the user was looking at and to anchor context evidence to source locations. This created a class of fragility:

- Line numbers change whenever any line above them is inserted or deleted.
- Cursor positions are ephemeral and do not survive across sessions.
- Prior-turn offsets stored in memory become stale as files evolve.
- File paths change when modules are renamed or moved.

When context evidence was anchored to coordinates that had drifted, the system produced misleading or irrelevant context: wrong evidence was surfaced, wrong entities were queried in the graph, and rendered reasoning packets referred to code that was no longer at the stored position.

The deeper problem was that the system was anchoring to *where* the user was rather than *what* the user was looking at. Semantic meaning is stable across refactors; coordinates are not.

**Constraints:**
- The VS Code symbol provider is available and can return the containing symbol for any cursor position.
- The DreamGraph knowledge graph holds canonical IDs for features, workflows, ADRs, and UI registry entries.
- Reasoning packets, evidence selection, and prompt assembly all depend on anchor quality.
- ADR-043 requires the context builder to use a plan → resolve → packet → render pipeline.

## Decision

Semantic anchors are the canonical identity for all context references in DreamGraph. Line numbers, cursor positions, and prior offsets are retained as advisory data only — they may be used as operational hints (e.g. to pass to a VS Code API) but must never be the primary identity for an anchor, evidence item, or graph lookup.

### Anchor promotion hierarchy (binding)

1. **Graph entity ID** — a `canonicalId` matched to a DreamGraph feature, workflow, ADR, or UI registry entry. Highest confidence. Survives all refactors.
2. **Symbol path** — a VS Code symbol provider path such as `ContextBuilder.buildEnvelope`. Survives line changes. Degrades gracefully when symbols are moved.
3. **File path + symbol name** — secondary identity when a full symbol path is not resolvable.
4. **Heuristic excerpt** — a short prose fingerprint derived from visible code context. Fallback only.
5. **Line number / cursor position** — advisory. Used only to seed VS Code API calls; never stored as identity.

### Migration policy (binding)

When a stored `SemanticAnchor` is re-evaluated against the current state of the codebase:

| Outcome | `migrationStatus` | Action |
|---|---|---|
| Symbol path matches exactly | `native` | Confidence boost |
| Leaf symbol name matches but path changed | `rebound` | Update `symbolPath`; keep identity |
| Fuzzy/partial name match only | `drifted` | Apply confidence penalty; flag as approximate |
| No match at all | `archived` | Set `historical: true`; exclude from live evidence |

### Anchor derivation order (binding)

For every new anchor derived during `buildEnvelope`:

1. Ask the VS Code symbol provider for the deepest symbol containing the position.
2. If found, build a `symbolPath` and emit a `SemanticAnchor` with `source: "symbol_provider"`.
3. After graph context is resolved, attempt graph promotion: match `symbolPath` or `label` against feature IDs, workflow IDs, ADR IDs, and UI registry IDs. If matched, set `canonicalId`, `canonicalKind`, and `migrationStatus: "promoted"`.
4. If symbol provider returns nothing, fall back to heuristic excerpt with `source: "heuristic"` and `approximate: true`.
5. Line and cursor coordinates are retained in the envelope for operational use but are not part of anchor identity.

### Alternatives Considered

- **Keep coordinate-first anchoring with heuristic correction** — Rejected: This permanently couples anchor quality to file stability and requires constant correction as files evolve.
- **Use only file path as anchor identity** — Rejected: File-level anchors are too coarse for evidence selection and do not survive module renames.
- **Promote to graph IDs only at display time** — Rejected: Late promotion means intermediate pipeline stages (evidence selection, token budgeting) use weaker anchors, producing lower-quality context packets.

## Consequences

- Context evidence is anchored to stable semantic identity rather than volatile coordinates.
- Prior-session anchors that have drifted are detected and handled (rebound, drifted, archived) rather than silently used.
- The `resolveAnchorMigration` API provides a single entry point for re-evaluating any stored anchor against the current symbol tree and graph.
- Rendering surfaces (context inspector, reasoning packet output) show promotion status, making anchor quality observable.
- Commands that receive prior-session anchors must call `resolveAnchorMigration` before using those anchors for evidence selection or graph lookup.

**Risks accepted:**
- Symbol provider calls add async latency to envelope construction.
- If the graph has sparse coverage, few anchors will be fully promoted to graph entity IDs; they will remain at symbol-path level.
- Migration re-evaluation adds complexity to the anchor lifecycle.

## Guard Rails

> ⚠️ Do NOT use raw line numbers or cursor positions as the primary identity for any anchor, evidence item, or graph lookup. They are advisory only.

> ⚠️ Do NOT store an anchor without at minimum a `symbolPath` or a `canonicalId`. A label-only anchor is a regression.

> ⚠️ Do NOT bypass the anchor promotion pass in `buildEnvelope`. Graph context must be resolved before anchors are finalised so promotion can run in one pass.

> ⚠️ Do NOT re-evaluate a stored anchor without calling `resolveAnchorMigration`. Silently using a stale anchor is the failure mode this ADR prevents.

> ⚠️ Do NOT demote a `promoted` or `native` anchor back to coordinate-based identity during serialisation or rendering.
