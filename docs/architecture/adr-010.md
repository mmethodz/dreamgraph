# ADR-010: Canonical Feature Identity Uses feature_* IDs

**Date:** 2026-04-17T17:25:03.842Z  
**Status:** accepted  
**Decided by:** system  
**Affected entities:** feature_ui_registry, feature_dashboard_server, feature_vscode_extension, dashboard, dashboard_server, server_dashboard, vscode_extension, extensions_vscode  

## Context

The fact graph contains duplicate and legacy feature identities for the same architectural areas, especially dashboard and VS Code extension surfaces (for example feature_dashboard_server vs dashboard/dashboard_server and feature_vscode_extension vs vscode_extension/extensions_vscode). This fragments ownership, weakens UI registry used_by mappings, and makes graph queries ambiguous.

**Constraints:**
- Must preserve existing graph data without destructive deletion
- UI registry ownership should resolve to stable feature IDs
- Changes must remain within existing DreamGraph feature/data protection model
- No accepted ADR currently defines canonical naming for feature identities

## Decision

Use feature_* IDs as the canonical identity scheme for authoritative feature entities. Legacy non-canonical feature entries may remain temporarily for historical continuity, but all new enrichment and UI registry used_by mappings must target canonical feature_* IDs. Consolidation work should strengthen canonical entries and mark legacy aliases as superseded in descriptions/metadata rather than relying on them for ownership.

### Alternatives Considered

- **Keep mixed historical IDs indefinitely** — Rejected: Leaves graph queries ambiguous and preserves fragmented ownership semantics.
- **Delete legacy feature entries immediately** — Rejected: Higher risk because other graph references may still point at those IDs and DreamGraph lacks a dedicated feature-delete/supersede flow.
- **Use source-path-derived IDs as canonical feature identifiers** — Rejected: Would be less stable for conceptual features spanning multiple files and UIs.

## Consequences

- UI registry ownership becomes consistent across dashboard and extension surfaces
- Future enrichment converges on stable canonical feature identifiers
- Legacy aliases can still be discovered during transition but should no longer own semantics

**Risks accepted:**
- Some existing queries may still return legacy features until broader cleanup is complete
- Without a dedicated deprecation mechanism for features, legacy nodes remain visible and may confuse users temporarily

## Guard Rails

> ⚠️ Do NOT introduce new authoritative feature entities for existing capabilities unless they follow the feature_* canonical ID scheme.

> ⚠️ Do NOT register UI used_by ownership against legacy aliases when a canonical feature_* entity already exists.

> ⚠️ Do NOT treat dashboard, dashboard_server, server_dashboard, vscode_extension, or extensions_vscode as authoritative owners while canonical feature_* entries exist.
