# ADR-049: LLM request timeout handling in VS Code architect chat

**Date:** 2026-04-21T22:00:14.862Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_agentic_tool_execution_loop, feature_architect_tool_call_translation, feature_vscode_extension, workflow_architect_agentic_tool_execution, workflow_vscode_assisted_reasoning  

## Context

Long-running LLM-initiated operations in the VS Code architect chat could stop in ways that were not surfaced clearly enough to the user, and command execution timeouts needed a consistent recovery policy. We needed an explicit, bounded timeout handling strategy that improves resilience without creating hidden infinite retries or masking persistent failures.

**Constraints:**
- Must stay within the VS Code extension host and architect chat execution path boundaries.
- Must preserve explicit timeout awareness rather than silently continuing as if the request succeeded.
- Must use bounded recovery only: no unbounded automatic retries.
- Must keep user trust by surfacing failure after automatic recovery is exhausted.
- Must remain compatible with existing architect agentic tool execution and VS Code assisted reasoning workflows.
- Must avoid introducing behavior that conflicts with existing accepted ADR guard rails; no conflicting accepted timeout ADR was found.

## Decision

Adopt a two-stage timeout recovery policy for VS Code architect-chat initiated operations: when a timeout is detected in the local command execution path, automatically retry exactly once with a longer timeout, then surface the failure to the user if the retry also fails. Timeout and cancellation signals must be treated as first-class outcomes and reported explicitly rather than being mistaken for successful completion or silent termination.

### Alternatives Considered

- **Fail immediately on first timeout with no retry** — Rejected: Too brittle for legitimate long-running operations and creates unnecessary user interruptions for recoverable cases.
- **Retry indefinitely or multiple times with increasing backoff** — Rejected: Risks hiding persistent failures, wasting resources, and reducing user trust through opaque automation.
- **Treat timeout as a generic error without explicit timeout semantics** — Rejected: Does not address the original issue of timeout awareness and makes diagnosis and recovery less clear.

## Consequences

- Long-running command-backed operations get one bounded self-healing attempt before user escalation.
- Timeouts become explicit observable outcomes in the architect chat path instead of silent collapses.
- Users retain control after the single retry fails, improving predictability and trust.
- Telemetry and future recovery logic can build on a clear timeout outcome model.

**Risks accepted:**
- A second attempt increases total wait time before the user is consulted.
- A longer retry timeout may still be insufficient for some workloads.
- Different timeout sources (provider request timeout vs local tool timeout) may still require distinct implementation details even under one policy umbrella.

## Guard Rails

> ⚠️ Do NOT add unbounded or silent retry loops for LLM or tool timeouts without revisiting this ADR.

> ⚠️ Do NOT suppress timeout/cancellation reporting to the user after the single automatic retry is exhausted.

> ⚠️ Do NOT treat timed-out operations as successful completion in chat state, traces, or result summaries.

> ⚠️ Revisit this ADR before changing the retry count above one or making retry behavior provider-specific in a way that changes user-visible semantics.
