# ADR-011: Semantic UI Registry Includes Abstract Visual Language and Layout Contracts

**Date:** 2026-04-17T17:39:25.086Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** feature_ui_registry, feature_dashboard_server, feature_vscode_extension, data_model_ui_element  

## Context

The current semantic UI registry captures purpose, data contracts, interactions, implementations, and ownership, but it under-specifies visual styling and layout characteristics that are architecturally important for preserving consistent UX across dashboard and extension surfaces. Without abstract visual metadata, the registry cannot fully describe composition intent, emphasis hierarchy, spacing density, navigation chrome, or state-driven styling while remaining platform-agnostic.

**Constraints:**
- UI registry must remain semantic and platform-agnostic rather than storing raw CSS or framework-specific style props.
- Visual metadata must describe intent and layout contracts, not pixel-perfect implementation details.
- Existing registered UI elements should remain valid and be enrichable incrementally.
- Metadata should support both server-rendered dashboard HTML and VS Code extension surfaces.

## Decision

Extend the semantic UI registry convention to include abstract visual and layout metadata for UI elements. This metadata should describe visual role, density, emphasis, chrome level, layout pattern, alignment, hierarchy, responsive behavior, and state-driven styling semantics, while avoiding concrete CSS values or implementation-specific component props.

### Alternatives Considered

- **Keep registry purely behavioral and omit visual/layout metadata** — Rejected: Leaves an architectural blind spot where shared UX structure and visual intent are undocumented and hard to port or validate.
- **Store raw CSS/design-token/framework style props directly in the registry** — Rejected: Violates platform abstraction and would couple the registry to specific implementations rather than semantic intent.

## Consequences

- UI registry entries can better express cross-platform UX intent and composition rules.
- Migration and reconstruction tasks can preserve layout and visual hierarchy more accurately.
- Some existing UI elements will need incremental enrichment to adopt the expanded metadata model.

**Risks accepted:**
- Visual metadata may drift toward implementation detail if not governed carefully.
- Different contributors may interpret abstract styling categories inconsistently until conventions stabilize.

## Guard Rails

> ⚠️ Do NOT store raw CSS, Tailwind classes, or framework-specific style props as authoritative UI registry metadata.

> ⚠️ Do NOT replace semantic purpose/data-contract metadata with purely visual descriptions; visual metadata is additive.

> ⚠️ When registering or updating UI elements, prefer abstract layout and visual intent terms such as card grid, dense inspector, primary action emphasis, or embedded shell chrome over pixel-level specifications.
