# ADR-085: Route GPT-5.5 Architect OpenAI calls through the Responses API

**Date:** 2026-04-27T14:22:58.651Z  
**Status:** accepted  
**Decided by:** collaborative  
**Affected entities:** extensions/vscode/src/architect-llm.ts, ArchitectLlm, OPENAI_MODELS, _callOpenAI, _callOpenAIWithTools  

## Context

The Architect extension added GPT-5.5 as an OpenAI model option, but its OpenAI non-streaming and tool-call paths still used Chat Completions even though GPT-5.5 guidance recommends the Responses API for reasoning, tool-calling, and multi-turn workloads.

**Constraints:**
- Architect must preserve its existing provider abstraction and ArchitectResponse/ArchitectToolResponse contracts.
- Architect tool-calling must continue to expose ToolUseRequest entries to the existing tool loop.
- Existing OpenAI models that still use Chat Completions must not be regressed by the GPT-5.5 migration.
- GPT-5.5 Responses API requests should use API-level reasoning effort and text verbosity controls with safe defaults.

## Decision

For OpenAI Architect calls whose model slug starts with gpt-5.5, route non-streaming and tool-call paths to /responses via a compatibility adapter. Leave existing Chat Completions paths in place for other OpenAI models and preserve the existing streaming implementation for now.

### Alternatives Considered

- **Replace all OpenAI Architect calls with the Responses API immediately.** — Rejected: This would risk regressions for existing OpenAI models and streaming behavior beyond the scope of the GPT-5.5 migration.
- **Keep GPT-5.5 on Chat Completions because the existing abstraction already works for other OpenAI models.** — Rejected: This conflicts with the GPT-5.5 migration guidance for reasoning, tool-calling, and multi-turn use cases.

## Consequences

- GPT-5.5 Architect non-streaming calls use /responses and can set reasoning.effort and text.verbosity.
- GPT-5.5 Architect tool calls are adapted from Responses API function_call output items into the existing ToolUseRequest contract.
- Existing non-GPT-5.5 OpenAI models continue using the prior Chat Completions implementation.

**Risks accepted:**
- The Responses API item schema may evolve or differ for hosted tools/phase metadata not yet represented in ArchitectToolResponse.
- The current adapter is stateless and replays raw message/tool-result items rather than adopting previous_response_id state management.
- GPT-5.5 streaming remains on the existing OpenAI streaming path until a separate streaming migration is implemented.

## Guard Rails

> ⚠️ Do not route GPT-5.5 Architect reasoning/tool-call workloads back to Chat Completions without revisiting this ADR.

> ⚠️ Do not remove the Chat Completions fallback for existing OpenAI models unless those models have been tested against the Responses API path.

> ⚠️ Preserve the ArchitectToolResponse ToolUseRequest contract when changing Responses API parsing.
